As a software engineer, I have spent my career thinking about the design of stable systems, and one of the most fundamental features of a stable system is reproducibility. Reproducibility is also a core value of experimental research. Reproducibility creates trust. Trust in a product or platform. Trust in experimental results.

While Jupyter Notebooks are incredibly powerful for exploratory analysis or ad hoc research, I found the use of them for deeper research to be unsettling. A single variable change, a cell run out of order, or even visualizations created based on the wrong dataset could create inaccurate interpretations of the subject I was studying.

In my research, I want to be able to study models as they evolve over training, not their final output. This means that I need to be able to analyze models across multiple checkpoints, sometimes over one hundred per model. There was no way I was going to feel confident in my results by running experiments in notebooks. I needed something more stable. More reliably reproducible.

I have been iterating on a platform that allows me to encapsulate model analysis into three separate concerns: **training** - which generates static checkpoint files, **analyzers** - which convert checkpoint files into analysis artifacts, and **views** - which transform analysis artifacts into visualizations.

Within each concern, an act is performed once using centralized, testable, and encapsulated code. A model checkpoint is exported once using a shared class for all models that share the same architecture. Analysis artifacts are generated using Analyzer classes that are designed to run on model checkpoints agnostically. And visualizations are defined once in code available immediately across all models. In addition to supporting reproducibility, the run-once paradigm allows me to leverage longer running analysis functions such as PCA, since all of the cost is paid once upfront.

Another side effect of keeping each of these concerns separate is that it allows me to evolve them independently. If I decide that a model would benefit from more training time, all I need to do is re-run training with a new checkpoint schedule and re-run the analysis on the newly created checkpoints. If I want to add another visualization that uses an existing analysis artifact, I do not need to re-run any training or analysis artifact generation. And if I want to add a legend to an existing visualization or add support for an epoch-cursor, that change is made in one place with no need to re-run training or analysis.

This three-stage pipeline, from training to analysis to visualization, allows me to spend more time interpreting the analysis rather than worrying about whether inconsistencies have crept in. When I encounter surprising or unexpected results in the visualizations, they are more likely to be real results. And those surprises give me more leads to follow into deeper analysis of how models learn.
